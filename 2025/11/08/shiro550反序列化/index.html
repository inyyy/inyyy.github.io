shiro550反序列化漏洞分析

漏洞分析：

Shiro-550 漏洞的核心问题在于 Shiro 使用硬编码的 AES 加密密钥处理 rememberMe cookie。攻击者可以利用这个已知的密钥创建恶意序列化对象，然后将其作为 rememberMe cookie 发送给服务器。服务器在反序列化这个 cookie 时，会执行其中的恶意代码，从而导致命令执行漏洞

一：环境搭建

用tomcat8版本+shiro项目打包的war包即可启动

二:shiro550分析

用户名root，密码secret，shiro框架最大的特征就是rememberMe，需要勾选记住我选项，下一次直接自动登录，本意是为了方便用户登录，成功登陆后会返回cookie，里面就是用户对应的rememberMe字段，这个字段在后端代码经过了序列化，aes对应密钥的加密，base64编码，如果构造一个cookie，里面的rememberme字段是恶意的内容，服务器接收后进行base64解码，aes解密，反序列化就会触发命令执行，在这里面最重要的就是aes密钥，shiro550漏洞的核心原因就是密钥被硬编码在源码中，在shiro1.2.4版本以上密钥在代码中被移除，要求开发作者自己设置，不设置就动态生成，避免硬编码

def aes_enc(data):
BS = AES.block_size
pad = lambda s: s + ((BS - len(s) % BS) * chr(BS - len(s) % BS)).encode()
key = “kPH+bIxk5D2deZiIxcaaaA==”
mode = AES.MODE_CBC
iv = uuid.uuid4().bytes
encryptor = AES.new(base64.b64decode(key), mode, iv)
ciphertext = base64.b64encode(iv + encryptor.encrypt(pad(data)))
return ciphertext

def aes_dec(enc_data):
enc_data = base64.b64decode(enc_data)
unpad = lambda s: s[:-s[-1]]
key = “kPH+bIxk5D2deZiIxcaaaA==”
mode = AES.MODE_CBC
iv = enc_data[:16]
encryptor = AES.new(base64.b64decode(key), mode, iv)
plaintext = encryptor.decrypt(enc_data[16:])
plaintext = unpad(plaintext)
return plaintext

二：cc11链

shiro框架的依赖中有cc库，cc6链不受jdk版本限制，在jdk8中都能跑起来

这里直接用cc6链打报错了，无法加载数组Transfoemer[],查了一下shiro框架有自定义的类加载器，导致无法加载数组，那么我们就要创造一个不需要Transformer数组的链，cc2的字节码链就不需要数组

利用InvokeTransformer反射TemplatesImpl类的newTransformer方法来触发getTransletInstance方法，从而触发defineTransletClasess方法加载字节码，这个difine方法会加载字节码，加载字节码后再newinstance方法会触发初始化导致静态方法被执行，根据之前的学习，TemplatesIMpl类的_class参数要为空且默认为空，_name参数要不为空，随便反射传一个就行

要将cc6和cc2联系起来，这个key值就必须是TemplatesImpl实例

这个key值是在TiedMapEntry进行初始化的，到这里思路理清了，sink点用cc6的

BeanComparator.compare方法

在这个方法里面会调用cb库提供的PropertyUtils.getPrperty方法，如果其中的实例化对象传参为TemplatesImpl类，属性传为OutputProperties,其就回去调用该对象的getOutputProperties方法

这个方法又会触发字节码链的关键方法newTransformer，从而形成一条链，再定位入口处
