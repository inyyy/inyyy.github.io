java代码审计之sql注入

一：jdbc的sql注入

jdbc的sql有两中模式，一种是statement，一种是preparedstatement也就是预编译

statement的sql写法：

String checkUserQuery = “select userid from sql_challenge_users where userid = ‘“ + username_reg + “‘“;

Statement statement = connection.createStatement();

ResultSet resultSet = statement.executeQuery(checkUserQuery);

从sql语句可以看出来是直接进行拼接的，单引号闭合，直接拼接就是导致sql注入的绝对原因，用户可以采用万能钥匙闭合sql语句，拖库得到所有的数据，类似admin’ or ‘1’=’1,sql语句就会变成：

select userid from sql_challenge_users where userid = ‘admin’ or ‘1’=’1’

preparedstatement的sql写法:

var statement = connection.prepareStatement(“select password from sql_challenge_users where userid = ? and password = ?”);

statement.setString(1, username_login);
statement.setString(2, password_login);

var resultSet = statement.executeQuery();

预编译提前写好sql语句让数据库理解查找逻辑，利用占位符替代用户输入，将用户的输入变成纯数据，用户输入部分不会执行sql语法

如果采用preparedstatement还是直接拼接sql不采用占位符的话sql注入是依旧存在的

二：mybatis下的sql注入

mybatis是对jdbc的封装，更加方便，在mybatis中预编译采用#{}接受参数，${}的话就是直接对参数进行拼接，mybatis和jdbc的区别是他的sql语句是写在mapper接口里面的，也就是mapper.xml文件里面，在springboot框架里面是在资源文件夹里面

直接拼接写法：

//Mybatis mapper接口
@Select(“select * from users where username = ‘${username}’”)
List findByUserNameVuln01(@Param(“username”) String username);

//Mybatis
@GetMapping(“/mybatis/vuln01”)
public List mybatisVuln01(@RequestParam(“username”) String username) {
return userMapper.findByUserNameVuln01(username);
}

问题还是直接进行拼接，根据闭合方式进行闭合可以引发sql注入

#{}预编译写法：

@GetMapping(“/mybatis/sec01”)
public User mybatisSec01(@RequestParam(“username”) String username) {
return userMapper.findByUserName(username);
}

mapper层代码

//Mybatis
@Select(“select * from users where username = #{username}”)
User findByUserName(@Param(“username”) String username);

一些无法直接预编译的关键字：

一：in关键字

in关键字直接使用#{}会报错，但是也不能将其直接进行拼接，正确的做法是对其中的元素进行foreach遍历后再采用#{}读取参数

错误写法：

mapper层代码

从xml中可以看出直接对参数进行了拼接，这样仍然会导致闭合实现注入

正确的usermapper.xml中的写法

遍历每一个元素然后预编译写法接受参数，比如输入的是1,2,3,4,5，foreach遍历就会一次读取1然后#{1}的方式进行取值

二：like模糊查询关键字

像”select * from table where id like ‘%”+xxx+””%’”这样的sql语句是直接进行拼接的，存在sql注入，但是like关键字的%通配符是不能被预编译的，它不是sql语句的一部分而是like关键字的特殊语法，直接拼接存在的问题不止是sql注入，如果没有对用户输入的%进行转义，还会造成慢查询，造成dos

慢查询：

当like的%通配符在左边时会进行全表查询，占用cpu，数据库查询时会进行回表操作，导致I/O堵塞，当用户输入大量的%会指数级增加匹配难度和时间，严重会造成dos

正确的写法应该是先对用户输入进行转义，然后预编译

jdbc：

// 1. 拿到用户输入
String keyword = request.getParameter(“keyword”);

// 2. 【关键】转义关键字里的 % 和 ，让它们变成普通字符
keyword = keyword.replace(“%”, “\%”).replace(““, “\_”);

// 3. 拼接通配符（只有这里能加 %）
String likeValue = “%” + keyword + “%”;

// 4. 预编译（安全！）
String sql = “SELECT * FROM users WHERE password LIKE ?”;
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, likeValue); // 这里才是真正安全的预编译

mybatis：

xml映射文件中的写法：

#{}进行预编译，mybatis会自动对%和_进行转义，避免了慢查询

错误写法：

SELECT * FROM users WHERE username LIKE ‘%${keyword}%’

用户输入‘ or 1=1 – 会直接进行脱库

三：order by ，from，group by关键字无法预编译

sql语法结构，表名，列名，order by ，fromshi无法进行预编译的，预编译的原理是先编译好sql语法结构让数据库理解查询逻辑，再传入纯数据，而order by 列名和from 表名都是sql语法结构的一部分，所以无法进行预编译，只能做好过滤，比如白名单或者只能传入固定的枚举值

三：mybatis-plus中的sql注入

mybatis-plus是mybatis的增强版，底层是mybatis，也是目前最主流的开发工具，对于单表查询非常方便，不用写sql语句，多表查询用xml更加方便，在mp中核心是条件构造器，querywrapper，可以用它拼接任意sql条件

比如简单查询：

// 查询 username 包含 “张三” 且 age > 18 的用户
QueryWrapper wrapper = new QueryWrapper<>();
wrapper.like(“username”, “张三”)
.gt(“age”, 18);

List users = userMapper.selectList(wrapper);

lambda表达式：

lambdaQuery()**
.like(User::getUsername, “张三”)
.gt(User::getAge, 18)
.list();**

1：last方法造成的sql注入

在MP里面last方法非常的霸道，它可以无条件的将last方法里面的语句拼接到预定义好的sql语句中，别的方法在传入时会将参数部分用？占位符替代，但是last方法是直接将输入拼接到sql语句中，所以它触发sql注入的原因就是绕过了MP底层的预编译机制

@RequestMapping(“/mybatis_plus/mpVuln03”)
public List mpVuln03( String id) {
QueryWrapper wrapper = new QueryWrapper<>();
wrapper.last(“order by “ + id);
return employeeMapper.selectList(wrapper);
}

直接将参数id进行拼接，比如：select * from table where id=’”+id+”‘

用户输入’ or 1=1 – 即可，还可以再后面拼接; 进行多重sql语句执行，达到堆叠注入

2：apply方法

apply方法是MP里面直接用来拼接原生sql的方法

  • 对比

    ：

    • ✅ 安全：wrapper.apply("name = {0}", userName); (自动处理特殊字符，防止注入)
    • ❌ 危险：wrapper.apply("name = '" + userName + "'"); (直接拼接字符串，存在注入风险)

apply方法支持占位符替换，{0},在MP中底层会将其理解为？占位符，使用preparedstatement处理，如果使用apply方法直接拼接会存在sql注入

代码实例

@RequestMapping(“/mybatis_plus/mpVuln01”)
public Employee mpVuln01(String name, String id) {
QueryWrapper wrapper = new QueryWrapper<>();
wrapper.eq(“name”,name).apply(“id=”+id);
Employee employee = employeeMapper.selectOne(wrapper);
return employee;
}

这里的name使用了预编译模式，是安全的，但是id是直接拼接的，存在sql注入，但是这个代码万能钥匙是打不通的，因为用的是MP的selectone方法，这个方法的原理是先底层调用selectlist方法，然后检查返回的数据是否等于1，等于1才会返回，等于0结果是null，大于1直接报错，所以这里使用报错注入，只有一行报错返回结果，?id=1%20and%20extracvalue(1,concat(0x7e,(select database()),0x7e)),此时我们传入的xpath表达式是databasename，而xpath表达式要求路劲必须以/开头，从而不符合爆出错误信息

3：exists/notExists 拼接产生的SQL 注入

// 直接拼接字符串
exists(String existsSql)
notExists(String notExistsSql)

exists和notExists方法不会对拼接的sql进行预编译处理，而是直接将其拼接到原生sql语句上

代码示例

@GetMapping(“/vuln_exists”)
public List vulnExists(String id) {
QueryWrapper wrapper = new QueryWrapper<>();

// 危险代码：直接拼接用户输入的 id
// 这里的 id 是 HTTP 请求参数，完全可控
wrapper.exists("select id from user where id = " + id);

return userMapper.selectList(wrapper);

}

直接拼接，notExists同理

解决方法：可以使用apply函数来实现exists方法的安全查询，因为apply方法是支持占位符预编译的，避免用户直接拼接

@GetMapping(“/safe_exists”)
public List safeExists(String id) {
QueryWrapper wrapper = new QueryWrapper<>();

// 使用 apply 配合占位符 {0}
// MP 会自动将 {0} 替换为 ? 并进行预编译
wrapper.apply("exists (select id from user where id = {0})", id);

return userMapper.selectList(wrapper);

}

4：having语句

having语句的作用是在分组后用于设置条件过滤组，where的作用是用于在分组前过滤行

@RequestMapping(“/mybatis_plus/mpVuln06”)
public List mpVuln06( String id) {
QueryWrapper wrapper = new QueryWrapper<>();
wrapper.select().groupBy(“id”).having(“id >” + id);
return employeeMapper.selectList(wrapper);
}

问题还是直接拼接

5：order by和group by语句

这两个语句后面接的是sql的语法结构部分所以是不能进行预编译的，所以要做好过滤，白名单或者过滤关键字，这两个存在注入的话常用报错注如和时间盲注

6：insql和notinsql

在 MyBatis-Plus (MP) 中，inSql 和 notInSql 是用于构建 子查询 的条件构造方法。它们专门用于处理 WHERE column IN (SELECT ...) 或 WHERE column NOT IN (SELECT ...) 的场景。

危险写法

String roleCode = “ADMIN’ OR ‘1’=’1”; // 恶意输入

// 危险：直接拼接字符串
wrapper.inSql(“id”, “SELECT user_id FROM user_role WHERE role_code = ‘“ + roleCode + “‘“);

第一个参数是列名，著查询的字段名，第二个参数是子查询语句，这里直接进行拼接

安全写法

String roleCode = userInput; // 用户输入，不可信

// 安全：使用 {0} 占位符，MP 会进行预编译处理
wrapper.inSql(“id”, “SELECT user_id FROM user_role WHERE role_code = {0}”, roleCode);

使用占位符预编译处理

7：MP中的innerinterceptor的sql注入

在mp中有一个interceptor和很多个innerinterceptor，interceptor是一个拦截器总管，而innerinterceptor就是下面分发的各个接口，分别负责各自具体的工作，其中插件分页拦截器存在sql注入，这主要是因为sql中的orderby字段和limit字段无法通过预编译，不能无条件的信任用户输入的排序字段

package com.drunkbaby.config;

import com.baomidou.mybatisplus.annotation.DbType;
import com.baomidou.mybatisplus.extension.plugins.MybatisPlusInterceptor;
import com.baomidou.mybatisplus.extension.plugins.inner.PaginationInnerInterceptor;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class MybatisPlusConfig { 

/**

  • 注册插件
    */
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() { 

    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

    // 添加分页插件
    PaginationInnerInterceptor pageInterceptor = new PaginationInnerInterceptor();
    // 设置请求的页面大于最大页后操作，true调回到首页，false继续请求。默认false
    pageInterceptor.setOverflow(false);
    // 单页分页条数限制，默认无限制
    pageInterceptor.setMaxLimit(500L);
    // 设置数据库类型
    pageInterceptor.setDbType(DbType.MYSQL);

interceptor.addInnerInterceptor(pageInterceptor);
return interceptor;
}

}

代码示例：

// 前端传来的 pageVo 包含 sortField (排序字段) 和 sortOrder (排序方式)
Page page = new Page<>(pageVo.getPageNum(), pageVo.getPageSize());
// 危险点：直接将前端传来的字段拼接到 SQL 中
page.addOrder(OrderItem.of(pageVo.getSortField(), “asc”.equals(pageVo.getSortOrder())));

// 执行查询
IPage result = userMapper.selectPage(page, wrapper);

MP进行的处理：MP的分页拦截器会解析这个page对象，addorder方法直接拼接生成一下类似的sql语句

SELECT * FROM user ORDER BY [前端传来的字段] ASC LIMIT 10

如果用户传来1，extractvalue(1,concat(0x7e,(select database()),0x7e))就会泄露数据库信息，这里通常是盲注形式，因为是分页查找，返回的数据量被size定死了，超出范围会报错

四：Hibernate框架中的sql注入

Hibernate是一个对jdbc进行封装的框架，编程的核心思想是对象关系映射ORM，它更加方便开发者对于数据库的操作，不用写sql语句，会自动将对象实例insert到表中，当查询时又会自动将数据库结果转换为java对象，Hibernate可以使用hql来执行sql语句，也可以直接执行sql语句，但是这两种方法都有可能存在sql注入

危险代码示例：

String username = request.getParameter(“username”);
String password = request.getParameter(“password”);

// 危险：直接拼接字符串
String hql = “FROM User WHERE username = ‘“ + username + “‘ AND password = ‘“ + password + “‘“;
Query query = session.createQuery(hql);
User user = (User) query.uniqueResult();

直接接受用户输入参数然后进行拼接，存在sql注入，直接闭合就可能脱库

正确做法

1：进行参数化查询，使用占位符，比如：username或者？替代

String username = request.getParameter(“username”);
String password = request.getParameter(“password”);

// 安全：使用占位符 :username 和 :password
String hql = “FROM User WHERE username = :username AND password = :password”;
Query query = session.createQuery(hql);

// 使用 setParameter 绑定参数
query.setParameter(“username”, username);
query.setParameter(“password”, password);

User user = (User) query.uniqueResult();

Hibernate中的原生sql写法：

Query query = session.createNativeQuery(“select * from user where username = ‘“ + username + “‘ and password = ‘“ + password + “‘“);

直接拼接

正确写法（设置参数）：

String parameter = “g1ts”;
Query query = session.createNativeQuery(“select * from user where name = :name”);
query.setParameter(“name”,parameter);
