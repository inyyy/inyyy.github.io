RMI的攻击方式

一：先创建一个远程接口，里面定义一个sayhello方法

这个接口要继承Remote类，并且里面的方法要抛出异常

二：创建这个接口的实现类Impl

这个类要继承自UnicastRemoteObject类，用于生成stub(存根)和skelet(骨架),是后续通信原理的重要步骤

该类的构造方法要抛出异常

该类是之前远程接口的实现类，要对接口方法进行重新

三：注册远程对象

第一行实例化一个远程对象，将实现了远程接口方法的实现类实例化

第二行创建一个注册中心

第三行将远程对象通过bind绑定到RMI注册表，bind的第一个参数是String形式，客户端通过这个字符串搜索，比如registry.lookup(),第二个参数是Remote对象，也就是远程对象

四：创建客户端

先获取注册中心的ip和端口

再通过远程对象的名字查找远程对象

最后便可调用

五：客户端打注册中心

list，bind，unbind，rebind，lookup方法，前面学习过了，因为list方法对应的case里面是没有反序列化的，所以list是无法进行攻击的

使用bind进行攻击

bind对应的case是有反序列化的，这里可以看到进行反序列化的是参数名和远程对象，所以可以把bind方法当作反序列化漏洞的入口，结合前面的cc链，如果目标服务端存在cc库相关组件，这里为了学习直接导入cc库，这里先学习如何利用cc1链攻击

AnnotationInvocationHandler#readObject(ObjectInputStream)
-> AbstractInputCheckedMapDecorator#entrySet()
-> AbstractInputCheckedMapDecorator$EntrySet#iterator()
AbstractInputCheckedMapDecorator$EntrySetIterator#EntrySetIterator(Iterator,AbstractInputCheckedMapDecorator)
-> AbstractInputCheckedMapDecorator$EntrySetIterator#next()
AbstractInputCheckedMapDecorator$MapEntry#MapEntry(Map.Entry, AbstractInputCheckedMapDecorator)
-> AbstractInputCheckedMapDecorator$MapEntry#setValue(Object)
TransformedMap#checkSetValue(Object)
InvokerTransformer#transform(Object) # 调用任意方法

前面了解过RMI的通信原理，客户端获得的远程对象引用本质上是一个java.lang.reflect.Proxy动态代理类，而不是真正的服务端对象

当客户端通过bind请求传递这个代理对象，服务端对这个代理对象进行处理的时候会去反序列化这个动态代理类

在这个类里面有一个方法newProxyInstance,这个方法会返回一个动态代理类，这个方法里面有三个参数，第一个是类加载器，第二个是要实现的接口类，第三个是代理调用的处理器，invoke方法就是在这里进行触发，当Proxy对象里面的任意方法被调用时都会去调用这个invoke方法，从而触发readObject方法

所以我们的思路就是构造一个恶意的Proxy对象，其必须实现Remote接口，因为RMI要求远程对象必须实现Remote接口，这样做才能满足要求，并且要利用Remote类的类加载器，cc1链的入口类就是继承自InvocationHandler类的，是满足要求的

poc：

Remote.class.cast():将Proxy代理类转换为Remote类，cast方法前是要转换的class类型

当RMI服务端处理这个远程对象时会对其进行反序列化，并且自动调用他的hashCode方法，toString方法，一旦调用了这个Proxy对象的方法就回去触发他的invoke方法，这里就是去触发invocationHandler类的invoke方法，反序列化这个对象时也会去触发readObject方法，两者都可以触发cc1链

使用rebind进行攻击

使用rebind攻击和bind差不多，将请求bind换成rebind就行

registry.rebind(“remotetest”,remote);

使用unbind和lookup进行攻击

先看unbind和lookup的源码部分有没有反序列化

两个都有反序列化部分，并且逻辑差不多，但是和前面的bind，rebind攻击不一样，lookup不能向服务端传递数据流，lookup的作用就是客户端获得服务端的远程对象，bind和rebind的设计就是客户端像服务端传递，如果这里直接lookup请求不能将恶意对象传递到服务端，它接受的是字符串，所以要从底层触发，强行改变lookup底层RemoteCall对象的输出流，绑定恶意对象并向服务端传递

lookup底层

高层表现：Remote obj = registry.lookup("test");

底层本质：一次基于 TCP 的 “请求 - 响应” 通信，包含「序列化请求包→网络传输→服务端反序列化处理→序列化响应包→客户端反序列化」全流程。

当我们调用registry.lookup(“test”)时客户端底层会执行这些操作

1：获取通信句柄

registry底层会持有UnicastRef对象，这个对象封装了服务端的ip，端口，通信协议(比如tcp协议)

2：创建远程调用对象

RemoteCall对象是RMI的数据包容器，里面包含了操作类型，比如lookup是2，操作标识，输出流(要传递的序列化的数据，就比如高层调用里面的序列化的参数test)

3：序列化请求参数

把lookup的参数序列化并保存到RemoteCall的输出流

4：发送请求包

把序列化后的数据发送到服务端地址

5：接着来到服务端操作

服务端收到tcp数据包后会对其进行反序列化，读取tcp数据包，解析出里面的RemoteCall对象，反序列化它的输出流(比如test参数)

然后会执行lookup逻辑，根据参数test查找注册表，获得对应的远程对象引用，然后把查到的远程对象引用序列化到响应包，通过tcp回传

6：客户端处理响应包

客户端拿到这个响应包，反序列化得到其中的远程对象，lookup返回这个远程对象引用，得到Remote obj

六：服务端打客户端

服务端打客户端大致分为两种情况，服务端返回Object对象和远程加载对象

服务端返回Object对象

在RMI中远程方法调用不止会返回基本数据类型，还可以返回Object对象，当服务端向客户端返回一个序列化的对象时，客户端接收后要对其进行反序列化，伪造一个服务端，调用远程方法后返回一个恶意对象给客户端就可以达到攻击目的

先自定义一个继承了Remote的接口，这个接口里面的getUser方法返回一个对象，我们可以写一个类实现这个接口的getUser方法，返回一个恶意对象

写一个ObjectReturn类实现getUser方法

这个类要继承UnicastRemoteObject类，因为在RMI中，客户端不能直接调用远程JVM的对象，必须将对象导出为远程对象，这个远程对象会监听网络连接，接收方法调用请求，反序列化参数，调用本地方法，序列化返回值并发送回客户端。

服务端

服务端先得到一个registry实例，实例化一个User对象，将其绑定到注册中心，以便后续客户端请求这个远程对象引用

客户端

调用getUser方法向客户端返回一个序列化的恶意对象，客户端接收到后对其进行反序列化触发后续攻击链

在这个过程中遇到的问题

在获得一个恶意对象时，最开始使用的是一层代理，但是客户端反序列化时触发了LazyMap.entrySet，但是没有触发LazyMap的get方法，导致攻击链没有被触发

原因：

  • 如果 AnnotationInvokeHandler的memberValues是原始 LazyMap，某些 JDK 实现可能会：
    • 缓存 entrySet() 视图；
    • 或在序列化时已经计算好 entries；
    • 不会在反序列化时重新调用 get()

双层代理又是如何触发get方法的：

客户端反序列化–>进入入口类，proxyhandler.readObject,因为开始的membervalues是proxymap，进入入口类的readObject方法是proxymap.entrySet，对proxymap进行遍历–>但是proxymap是一个动态代理类，调用proxymap.entrySet会触发他的处理器的invoke方法，handler.invoke(proxyMap, “entrySet”, null)

handler.invoke的执行逻辑

public Object invoke(Object proxy, Method method, Object[] args) {
if (“entrySet”.equals(method.getName())) {
return this.memberValues.entrySet(); // ← memberValues 是 lazyMap!
}
// …
}

所以handler的invoke方法返回的是lazyMap.entrySet

接着handler.readObject拿到entrySet后进行遍历

for (Map.Entry e : entrySet) {
String key = e.getKey();
Object value = e.getValue(); // 关键
}

触发了getValue方法从而触发了get方法，最后就触发了攻击链
